Queries ‚Äî Security

Since a GraphQL endpoint provides arbitrary access to your application, you should employ safeguards to prevent large queries from swamping your system.

Limiting lists of items

Always limit the number of items which can be returned from a list field. For example, use a limit: argument and make sure it‚Äôs not too big:

1field :items, types[ItemType] do
2  argument :limit, types.Int, default_value: 20
3  resolve ->(obj, args, ctx) {
4    # Cap the number of items at 30
5    limit = [args[:limit], 30].min
6    obj.items.limit(limit)
7  }
8end

This way, you won‚Äôt hit your database for 1000 items!

Enforce a timeout

You can apply a timeout to query execution with TimeoutMiddleware. For example:

1MySchema.middleware << GraphQL::Schema::TimeoutMiddleware.new(max_seconds: 2)

After max_seconds, no new fields will be resolved. Instead, errors will be added to the errors key for fields that weren‚Äôt resolved.

Note that this does not interrupt field execution. If you‚Äôre making external calls (eg, HTTP requests or database queries), make sure to use a ‚Äúlower level‚Äù timeout for the specific operation.

To log the error, pass a block to the middleware:

1MySchema.middleware << GraphQL::Schema::TimeoutMiddleware.new(max_seconds: 2) do |err, query|
2  Rails.logger.info("GraphQL Timeout: #{query.query_string}")
3end

Prevent complex queries

Fields have a ‚Äúcomplexity‚Äù value which can be configured in their definition. It can be a constant (numeric) value, or a proc. It can be defined as a keyword or inside the configuration block. For example:

 1# Constant complexity:
 2field :top_score, types.Int, complexity: 10
 3
 4# Dynamic complexity:
 5field :top_scorers, types[PlayerType] do
 6  argument :limit, types.Int, default_value: 5
 7  complexity ->(ctx, args, child_complexity) {
 8    if ctx[:current_user].staff?
 9      # no limit for staff users
10      0
11    else
12      # `child_complexity` is the value for selections
13      # which were made on the items of this list.
14      #
15      # We don't know how many items will be fetched because
16      # we haven't run the query yet, but we can estimate by
17      # using the `limit` argument which we defined above.
18      args[:limit] * child_complexity
19    end
20  }
21end

Then, define your max_complexity at the schema-level:

1MySchema = GraphQL::Schema.define do
2 # ...
3 max_complexity 100
4end

Or, at the query-level, which overrides the schema-level setting:

1MySchema.execute(query_string, max_complexity: 100)

Using nil will disable the validation:

1# üòß Anything goes!
2MySchema.execute(query_string, max_complexity: nil)

To get a feeling for complexity of queries in your system, you can use the QueryComplexity query reducer. Hook it up to log out values from each query:

1log_query_complexity = GraphQL::Analysis::QueryComplexity.new { |query, complexity| Rails.logger.info("[GraphQL Query Complexity] #{complexity}  | staff? #{query.context[:current_user].staff?}")}
2MySchema.query_analyzers << log_query_complexity

Prevent deeply-nested queries

You can also reject queries based on the depth of their nesting. You can define max_depth at schema-level or query-level:

1# Schema-level:
2MySchema = GraphQL::Schema.define do
3  # ...
4  max_depth 10
5end
6
7# Query-level, which overrides the schema-level setting:
8MySchema.execute(query_string, max_depth: 10)

You can use nil to disable the validation:

1# This query won't be validated:
2MySchema.execute(query_string, max_depth: nil)

To get a feeling for depth of queries in your system, you can use the QueryDepth query reducer. Hook it up to log out values from each query:

1log_query_depth = GraphQL::Analysis::QueryDepth.new { |query, depth| Rails.logger.info("[GraphQL Query Depth] #{depth} || staff?  #{query.context[:current_user].staff?}")}
2MySchema.query_analyzers << log_query_depth

Only execute predefined queries

If you don‚Äôt want to accept arbitrary queries from the ‚Äúoutside world‚Äù, you can cache queries on the server and fetch them in response to specific requests.

For example, you could store parsed GraphQL::Language::Nodes::Document objects in a cache:

1parsed_document = GraphQL.parse(query_string)
2operation_name = parsed_document.definitions.first.name
3MyCache.set(operation_name, parsed_document)

Then, later, you could fetch the document from storage and use it to run a query:

 1# later ...
 2operation_name = params[:operation_name]
 3document = MyCache.get(operation_name)
 4if document.nil?
 5  raise("No stored operation called #{operation_name}")
 6else
 7  # use `document` instead of a query string:
 8  MySchema.execute(document, context: { ... })
 9end

This way, no unknown queries are evaluated by the server.
